# RAG 项目笔记

RAG 的主流程是先从外部知识库中检索相关证据，再把证据和用户问题一起放入 prompt，让大语言模型基于证据生成回答。

一个企业知识库问答系统通常包含离线建库和在线问答两部分。离线阶段需要文档解析、文本清洗、chunk 切分、embedding 编码和索引构建。在线阶段需要 query 检索、候选重排、上下文拼接、模型生成、引用输出和后处理。

chunk 太小会导致语义不完整，答案可能被切碎。chunk 太大会引入噪声，占用上下文窗口，并降低检索粒度。实际项目中通常需要比较不同 chunk size 和 overlap 对 Recall@k、回答正确率和延迟的影响。

BM25 擅长关键词、专有名词和编号匹配。向量检索擅长语义相似和同义表达。两者结合的 hybrid retrieval 通常比单独使用一种检索方式更稳定。

rerank 通常放在召回之后。向量检索或 BM25 先快速召回 top-20 或 top-50，然后 cross-encoder reranker 对 query 和候选 chunk 做更充分的交互建模，最后选择 top-5 放入 prompt。

为了降低幻觉，可以要求模型只基于上下文回答，输出引用来源；当检索结果置信度低或上下文没有答案时，系统应该拒答，例如输出“根据现有资料无法确定”。
